# UX/UI Patterns: 去除密码校验（本地个人部署模式）

> Generated: 2026-04-12

## Core User Flows

### Primary (Happy Path)

1. **首次启动（CLI / Web 初始化向导）**
   - 用户运行启动脚本（如 `start.sh`）
   - 脚本检测到数据库为空 / 无用户配置，进入首次设置流程
   - 提示："是否为本地访问启用密码保护？"
     - 选项 A：「启用密码保护」（推荐用于多用户设备或公开网络）
     - 选项 B：「直接访问，无需密码」（默认，适用于个人独立设备）
   - 若选择 B：记录 `auth_enabled=false`，后续所有请求绕过鉴权
   - 若选择 A：提示输入新密码，使用 Argon2 哈希存储，记录 `auth_enabled=true`
   - 启动服务，浏览器自动打开（或提示访问地址）

2. **常规访问（无密码模式）**
   - 用户打开 Web 应用
   - 前端检测到 `auth_enabled=false`，直接进入主界面
   - 无需登录页、无 JWT 获取流程

3. **常规访问（有密码模式）**
   - 用户打开 Web 应用
   - 前端重定向至登录页
   - 用户输入密码，后端 Argon2 校验成功后签发 JWT
   - 进入主界面

### Alternative Paths

| 路径 | 场景 | 处理方式 |
|------|------|----------|
| **从有密码切换为无密码** | 用户已在设置中启用密码，后反悔 | 设置页提供「关闭密码保护」开关，需当前密码确认（或二次确认），成功后清除哈希、设置 `auth_enabled=false` |
| **从无密码切换为有密码** | 用户初始未设密码，后期希望加固 | 设置页提供「开启密码保护」，强制输入并确认新密码，立即生效，当前会话保持登录（或要求重新登录以验证） |
| **密码重置（启动脚本）** | 用户忘记密码，或购买了二手 NAS/继承了配置 | 运行启动脚本附加参数/菜单项「重置密码」，明确警告「此操作将清空所有现有数据」，用户输入 `RESET` 或 `y/n` 二次确认后，删除数据库并回到首次启动流程 |
| **密码重置但取消** | 用户在确认阶段犹豫 | 提供明确的「取消」选项，返回主菜单或退出脚本，不做任何修改 |
| **非法访问尝试** | 有人在无密码模式下直接访问受保护端点 | 后端统一返回 401/403，前端根据错误码决定是否弹出登录框或提示「访问被拒绝」 |

## State Inventory

| State | Trigger | Recommended Pattern |
|-------|---------|---------------------|
| **Empty / First-run** | 数据库无用户记录，`auth_config` 缺失 | 显示一次性初始化向导（Wizard），用清晰的单选或 toggle 呈现「是否启用密码」，避免跳过按钮造成困惑 |
| **Loading** | 前端启动时向后端查询 `/api/auth/status` | 显示居中 spinner 或骨架屏，避免白屏；加载时间应 <300ms |
| **Direct Access** | `auth_enabled=false` | 直接渲染主应用，无登录页闪跳；URL 保持干净（无 `#login` 哈希） |
| **Login Prompt** | `auth_enabled=true` 且未持有有效 JWT | 极简登录页：密码输入框 + 提交按钮 + 可选「显示密码」toggle；无注册/找回密码入口（由 CLI 脚本负责重置） |
| **Login Error** | 密码错误、网络异常、服务端 500 | 密码错误：输入框下方红色提示「密码错误」；服务端错误：toast/alert 提示「服务暂时不可用，请稍后重试」 |
| **Login Success** | 密码校验通过，JWT 写入 | 平滑过渡动画（fade/slide）进入主界面，或直接从登录页切走 |
| **Settings - Toggle Auth** | 用户在设置页开关密码保护 | 开关即时弹窗确认；若关闭密码，需再次提示安全风险；操作成功后 toast 反馈 |
| **Reset Warning** | 用户在 CLI 中选择「重置密码」 | 高对比度警告文本，必须使用用户主动输入确认字符（如 `RESET`）或 `y/N`，避免误触 |
| **Reset Success** | 确认后清空数据并完成重置 | 明确输出「数据已清空，新密码已设置：xxx」或「请重新运行脚本设置新密码」 |

## UI Pattern Library

### 1. 单用户工具首选：Onboarding Wizard（首次启动向导）
- **来源参考**：Plex、Home Assistant、Immich
- **描述**：首次访问时不是登录页，而是 1-3 步的初始化向导。第一步即询问「是否为这台设备设置密码」。
- **为什么适合**：本地自托管工具没有「注册用户」概念，向导模式比空白登录页更自然。
- **视觉要点**：进度条或步骤指示器、单选卡片（Enable Password / No Password）、底部有「下一步」主按钮。

### 2. 可选鉴权的 Settings Toggle（设置中开关密码）
- **来源参考**：Tailscale admin console、Pi-hole
- **描述**：在「安全设置」中放置一个主开关 `Enable authentication`。打开时展开密码输入框；关闭时弹出二次确认模态框。
- **为什么适合**：允许用户在后期灵活调整，无需编辑配置文件。
- **视觉要点**：开关使用标准 toggle（iOS/Android 风格），状态变化即时反馈；关闭时模态框用橙色/黄色背景强调警告。

### 3. 极简本地登录页（Local Login Screen）
- **来源参考**：Jellyfin、Navidrome
- **描述**：仅保留密码输入（无用户名，因为是单用户）。背景可为纯色或应用品牌色，无冗余链接。
- **为什么适合**：减少认知负荷；用户知道这是自己的设备，不需要「注册」或「忘记密码」的 Web 流程。
- **视觉要点**：大字号标题 + 聚焦状态的密码输入框 + 显眼的登录按钮；Enter 键直接提交。

### 4. CLI 二次确认模式（Destructive Confirmation）
- **来源参考**：Docker CLI、`rm -rf` 保护、`git filter-branch`
- **描述**：在终端脚本中， destructive action 要求用户输入特定字符串（如 `RESET`）或颜色高亮的 `y/N` 来确认。
- **为什么适合**：启动脚本重置密码意味着数据全清，必须防止误触回车键。
- **视觉要点**：红色 ANSI 高亮「警告：这将删除所有数据！」；清晰列出将被删除的文件/数据库路径；默认选项为 `N`。

### 5. 自动跳过登录（Silent Auth Bypass）
- **来源参考**：qBittorrent Web UI（可选关闭鉴权后的行为）
- **描述**：一旦配置为无需密码，前端在初始化时直接获取免鉴权的 `auth_status`，立即进入主界面。
- **为什么适合**：避免用户每次看到登录页闪一下再消失的不良体验。
- **视觉要点**：初始化加载状态应足够快（或缓存配置到 localStorage），让用户无感知。

## Micro-interactions & Animations

| 场景 | 建议的动效 | 目的 |
|------|-----------|------|
| 登录页 → 主界面 | 淡出（fade out）登录页，同时淡入主界面内容（200-300ms） | 减少上下文切换的突兀感 |
| 密码输入错误 | 输入框轻微水平晃动（shake, 300ms）+ 边框变红 | 提供即时、直观的错误反馈 |
| 显示/隐藏密码 | 眼睛图标平滑切换（opacity 变化） | 明确告知当前可见性状态 |
| 设置页开关密码保护 | Toggle 滑动动画 + 下方表单区域高度展开/收缩（200ms ease） | 关联控件显隐，符合用户预期 |
| 登录按钮 loading | 按钮宽度不变，内部文字替换为 spinner，禁用状态变灰 | 防止重复提交，同时保持布局稳定 |
| CLI 警告输出 | 无需动画，但使用 ANSI 颜色（红/黄/绿）和空行分隔 | 在终端环境中快速抓住注意力 |

## Accessibility Requirements (WCAG 2.1 AA)

1. **焦点管理（Focus Management）**
   - 登录页加载后，焦点应自动落在密码输入框（`autofocus`）。
   - 设置页中开启/关闭密码保护的模态框弹出后，焦点应trap在模态框内；关闭后焦点回到触发按钮。

2. **颜色对比度**
   - 所有文字与背景对比度至少 4.5:1（小号文字）或 3:1（大号文字/图标）。
   - 错误提示不能仅依赖红色，应同时提供图标或文字说明（如「密码错误」）。

3. **表单标签与关联**
   - 密码输入框必须有 `<label>` 并通过 `for` 关联，或使用 aria-label。
   - 提交按钮需有明确的可访问名称（不是模糊的「Submit」）。

4. **状态变化通知**
   - 登录成功/失败、密码保护开关结果，需通过 `aria-live` 区域（polite）通知读屏软件，避免仅视觉反馈。

5. **键盘可访问性**
   - 整个登录和设置流程应完全支持键盘操作（Tab 切换、Enter 提交、Esc 关闭模态框）。
   - 无鼠标依赖的隐藏功能。

6. **错误预防（3.3.4）**
   - 关闭密码保护、重置密码等 destructive action 必须提供可逆的确认步骤，防止数据丢失。

## Platform Considerations

### Web / Desktop Browser
- **为主平台**：所有功能优先在此实现。
- 推荐 Responsive 布局：登录页在宽屏居中卡片（max-width 400px），窄屏全屏垂直居中。
- 考虑 PWA 场景：如果用户将应用添加到主屏幕，无密码模式下应像原生应用一样直接打开。

### Mobile Browser
- 登录页输入框在 iOS Safari 上需确保 `font-size >= 16px`，避免自动缩放。
- 密码可见性 toggle 尺寸不小于 44x44 CSS 像素（触摸目标）。
- 模态框纵向居中，小屏时改为底部 sheet 或全屏避免截断。

### CLI（启动脚本）
- 这是「重置密码」和「首次配置」的唯一入口，必须兼容标准 Linux/macOS/Windows 终端。
- 使用 `read -p` 或 Python `input()` 均可，但需处理非交互式环境（如 systemd service）的默认行为：非交互模式下自动选择「无密码」或读取环境变量。

## Anti-patterns to Avoid

1. **模糊的「跳过」按钮**
   - 不要在首次向导中使用「跳过」作为关闭密码的文案。用户不知道「跳过」的是什么。应明确写「无需密码，直接访问」。

2. **Web 端的「忘记密码」链接**
   - 本地个人部署场景下，没有邮件服务器，无法提供邮件找回。放置「忘记密码」链接只会导致死胡同。应引导用户「运行启动脚本选择重置密码」。

3. **无确认的数据删除**
   - 重置密码必须伴随「所有数据将被清空」的警告，且不能只靠一次 `y` 确认。至少要求输入 `RESET` 或两次 `y`。

4. **登录页先渲染再跳转**
   - 如果已知 `auth_enabled=false`，不要让登录页先出现 100ms 再 redirect。应在初始化 loading 阶段就完成判断。

5. **关闭密码保护无需确认**
   - 用户在设置页一键关闭密码保护属于安全降级，必须弹出确认框说明风险。

6. **隐藏的后门或硬编码密码**
   - 不要为了「方便」留一个 admin/admin 的默认密码。这违背了本地个人部署的安全信任边界。

## Recommended Approach

针对本项目的具体建议：

### 1. 首次启动：CLI 引导 + Web 向导双通道
- **CLI 优先**：`start.sh` 检测到首次运行时，在终端询问是否启用密码。这样即使无浏览器环境也能完成配置。
- **Web 备用**：若用户通过 docker/systemd 等方式跳过 CLI，Web 端 `/setup` 页面作为兜底，展示同样的选择。

### 2. 鉴权状态接口
- 后端新增 `GET /api/auth/status`，返回 `{ "auth_enabled": boolean }`。
- 前端应用启动时先请求该接口，根据结果决定渲染登录页还是主页。
- 该接口本身无需鉴权。

### 3. 关闭/开启密码的 Web 设置入口
- 在「系统设置」中增加「安全」分组。
- 使用标准 toggle 控制 `auth_enabled`。
- **开启**：展开密码输入 + 确认密码输入，后端校验两次一致后更新。
- **关闭**：弹出模态框，要求输入当前密码验证身份，验证通过后设置 `auth_enabled=false` 并清除密码哈希。

### 4. 重置密码：纯 CLI 流程
- 在 `start.sh` / `manage.py` 中增加菜单项：`[4] 重置密码（将清空所有数据）`。
- 终端输出红色高亮警告，列出将被删除的数据库文件路径。
- 要求用户输入大写 `RESET` 确认。
- 确认后删除数据库，重新走首次启动流程。

### 5. 默认策略
- **默认不启用密码**：降低个人用户的心理门槛和首次使用摩擦。
- 在首次向导和 README 中用文字提示用户：「如果您的设备可能被他人使用，建议启用密码保护」。
