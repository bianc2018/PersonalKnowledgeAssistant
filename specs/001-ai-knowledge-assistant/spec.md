# Feature Specification: AI 知识管理助手

**Feature Branch**: `001-ai-knowledge-assistant`  
**Created**: 2026-04-04  
**Status**: Draft  
**Input**: User description: "需要实现一个基于ai驱动的知识管理助手，支持为用户管理知识(类似知识库)，支持对话查询知识，支持基于用户需求调研领域知识，形成调研报告，支持基于网络或者常识评估知识的置信度。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 添加并管理个人知识 (Priority: P1)

用户可以将日常收集到的文档、笔记、网页链接等内容整理进一个统一的知识库，方便后续查找和使用。

**Why this priority**: 知识库管理是整个产品的核心基础能力。没有可靠的知识入库和管理，后续的查询、调研等功能都无从谈起。

**Independent Test**: 用户能够独立上传一段文本并保存，随后在知识库列表中查看到该内容，验证其标题和摘要是否正确展示。

**Acceptance Scenarios**:

1. **Given** 用户进入知识库管理页面，**When** 用户输入一段文本并确认保存，**Then** 系统将该内容保存为一条知识，并在知识库列表中可见。
2. **Given** 用户已保存多条知识，**When** 用户搜索关键词，**Then** 系统返回标题或内容匹配的相关知识列表。

---

### User Story 2 - 对话式查询知识 (Priority: P1)

用户通过自然语言与助手进行对话，系统基于用户个人知识库中的内容给出准确回答，并标注信息来源。

**Why this priority**: 对话查询是用户获取知识价值的最直接方式，是驱动用户持续积累知识的核心动机。

**Independent Test**: 用户在知识库中已保存一条关于某主题的内容后，用自然语言提问该主题，系统给出的回答必须引用该内容。

**Acceptance Scenarios**:

1. **Given** 知识库中已存在相关知识，**When** 用户以自然语言提问，**Then** 系统基于库中内容生成回答，并指明参考来源。
2. **Given** 知识库中不存在与用户问题相关的内容，**When** 用户提问，**Then** 系统告知用户当前知识库中未找到相关信息，而非编造答案。

---

### User Story 3 - 生成领域调研报告 (Priority: P2)

用户输入一个调研主题，系统以异步方式检索网络信息并综合生成一份结构化的调研报告。用户在调研过程中可查看实时进度和阶段性输出；当系统遇到需决策的问题时会暂停并向用户提问，待用户回复后继续生成，最终帮助用户快速了解陌生领域。

**Why this priority**: 相比被动查询已有知识，主动调研能够扩展用户知识边界，是产品差异化的重要能力。

**Independent Test**: 用户提交一个明确的调研主题（如"低空经济政策分析"），在调研过程中能查看进度；当系统遇到分支决策时页面向用户提问，用户给出决策后系统继续生成，最终返回一份包含背景、现状、趋势、结论等章节的报告。

**Acceptance Scenarios**:

1. **Given** 用户提交调研主题和范围描述，**When** 系统开始处理，**Then** 用户可在页面上查看调研进度及阶段性输出摘要。
2. **Given** 调研过程中系统遇到需要用户决策的节点，**When** 系统暂停并向用户展示决策选项，**Then** 用户做出选择后调研流程继续执行。
3. **Given** 调研完成后，**When** 用户选择将报告保存到知识库，**Then** 系统以知识条目形式存储该报告并支持后续对话查询。

---

### User Story 4 - 知识置信度评估 (Priority: P2)

系统对每个知识条目进行置信度评估，帮助用户识别可靠信息和可能存在疑问的内容。

**Why this priority**: 置信度评估能够提升用户对知识库内容的信任感，但属于增强型体验，可在核心功能稳定后逐步完善。

**Independent Test**: 用户上传一条带有明显事实性陈述的知识后，系统显示该知识的置信度评分及评估依据（如网络验证结果或常识一致性说明）。

**Acceptance Scenarios**:

1. **Given** 系统接收到新知识，**When** 置信度评估完成后，**Then** 该知识条目显示一个可视化的置信度评分（如高/中/低或百分比）。
2. **Given** 用户查看某条知识的详情，**When** 用户查看置信度信息，**Then** 系统展示评估依据的简要说明。

---

### Edge Cases

- 用户上传的知识内容为空或长度低于 5 个字符时，系统应在前后端进行校验并拒绝保存（参见 FR-001a）。
- 当调研主题过于宽泛或模糊时，系统先按最宽范围自动生成初步概要，再由 AI 主动询问用户是否合适（参见 FR-007a）。
- 本地服务未启动时，用户通过浏览器访问应收到明确的服务不可用提示。
- 网络不可用或外部搜索源受限时，本地知识库管理和查询保持可用；若配置了本地模型，调研可降级至本地模型执行并提示用户处于降级模式；外部服务恢复后，系统重新通过外部 AI 评估并推送更新后的调研报告（参见 FR-012 ~ FR-014）。
- 同一条知识被多次修改后，每个历史版本均独立保存其置信度评估结果；修改生成新版本时若内容变化 >20% 则自动重新评估，旧版本的评估记录保持不变（参见 FR-008）。
- 导入备份压缩包时若文件损坏或包含不支持格式，系统 MUST 跳过该文件并继续导入其余内容（参见 FR-011）。
- 知识库总容量随多媒体文件增长而扩大时，当超过用户配置的磁盘阈值，系统 SHOULD 自动对旧媒体文件进行归档压缩（如 gzip），以保证本地磁盘读写性能不显著下降（参见 FR-026）。
- 用户忘记密码时，本地加密数据不可恢复，系统应明确提示并引导用户重新初始化系统后导入已有备份（参见 FR-025）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 允许用户以文本、文件或网页链接的形式添加知识到个人知识库，支持图片、代码、视频、音频等多种格式，单文件大小上限为 1GB。System MUST 自动将非文本媒体内容（如图片 OCR、音视频转录、文档全文提取）转换为可搜索文本，以便支持检索和对话查询；当自动提取失败时，System MUST 保存原始文件并标记提取失败状态，同时尝试利用文件名、扩展名等元数据进行极简索引，并向用户明确提示该文件暂不支持内容级搜索与对话引用。
- **FR-001a**: System MUST 在前后端对知识内容进行最小长度校验：当用户输入的文本内容为空或纯文本长度少于 5 个字符时，拒绝保存并向用户提示补充有效内容。
- **FR-002**: System MUST 将用户添加的知识组织为可检索、可浏览的知识条目列表。
- **FR-003**: System MUST 支持自然语言对话，用户可通过提问查询个人知识库中的内容；System MUST 持久化保存完整对话历史，允许用户查看会话列表并继续之前的话题。RAG 检索采用向量搜索（sqlite-vec）和全文搜索（FTS5）并行执行：向量搜索召回 Top-15 相关片段，FTS5 召回 Top-15 相关片段，去重合并后按相似度/相关度分数加权排序，最终取 Top-10 作为生成回答的上下文。
- **FR-004**: System MUST 在回答问题时引用所依据的知识来源，无法找到依据时必须明确告知用户。
- **FR-005**: System MUST 基于用户对话历史提取用户的领域偏好和知识水平，在生成调研报告和回答问题时进行个性化调整（如已知领域可减少基础说明，未知领域需补充背景介绍）。UserProfile 更新触发条件为：每累计 5 轮对话后自动分析一次，或当用户首次提问一个此前未涉及的新领域时立即触发。
- **FR-006**: System MUST 允许用户提交一个调研主题，并在合理时间内生成结构化的调研报告；System MUST 通过 Server-Sent Events (SSE) 向页面实时推送调研进度及阶段性输出摘要。
- **FR-006a**: `privacy_settings.allow_web_search` 为外部网络搜索的总开关：关闭时禁止任何外部网络搜索（LLM 自带搜索、独立搜索 API、HTTP 爬虫均不可用）；开启时，System MUST 按以下优先级获取外部网络信息：(1) 若 `llm_config.enable_search` 为 true，优先使用 LLM 端点自带的搜索能力；(2) 当 LLM 端点不支持搜索或 `enable_search` 为 false 时，回退至用户配置的独立搜索 API（如 Tavily、SerpAPI 等兼容端点）；(3) 当以上均不可用时，使用系统内置的轻量 HTTP 爬虫直接抓取网页；System MUST 在界面向用户展示当前实际使用的搜索源类型。
- **FR-007**: 调研报告 MUST 综合多个来源的信息，包含背景、关键观点、趋势分析及结论等结构化章节。
- **FR-007a**: 当调研主题过于宽泛或模糊时，System MUST 先按最宽范围自动生成初步概要，再由 AI 主动询问用户是否合适；若用户反馈不合适，System MUST 提出 2-3 个细化维度的问题，待用户补充后重新聚焦并继续调研。
- **FR-008**: System MUST 在知识入库/更新时自动对每条知识进行置信度评分，并在界面上可视化展示；知识内容的任何修改均 MUST 生成新的 `KnowledgeVersion`；置信度重新评估仅在内容差异比例 >20% 或被用户主动请求时触发；System MUST 提供手动重新评估入口，允许用户随时触发；每次生成新版本时必须保存独立的置信度评估结果，旧版本的评估记录不覆盖。
- **FR-009**: 置信度评估 MUST 借助外部网络信息或常识推理进行验证，并给出简要依据。
- **FR-010**: System MUST 提供知识库导出功能，将当前用户的全部知识及相关元数据打包为 ZIP 压缩文件；压缩包内必须包含一份 JSON 元数据文件（描述知识条目、版本记录、目录结构及 embedding 模型标识），并将原始媒体附件按原路径或目录结构存放；导出包 MUST 不包含 sqlite-vec 生成的嵌入向量，以减小体积并避免跨模型兼容性问题。
- **FR-011**: System MUST 支持通过导入已导出的 ZIP 压缩文件，在新系统环境中恢复知识库，实现数据迁移；导入过程必须校验 JSON 元数据，并安全跳过损坏或不支持的文件，在导入完成后以汇总报告形式提示用户被跳过的文件及原因；当导入环境中 embedding 模型与元数据标识不一致或向量缺失时，System MUST 自动触发重新计算嵌入向量。
- **FR-012**: 当外部 AI/搜索服务不可用时，System MUST 保持本地知识库管理和对话查询功能可用，并向用户明确提示调研和置信度评估能力受限。
- **FR-013**: 若用户配置了本地模型，System SHOULD 在外部服务不可用时将调研功能降级至本地模型执行，并在界面上提示用户当前处于降级模式。
- **FR-014**: 当外部服务从不可用恢复为可用后，System SHOULD 自动将处于 `pending_recheck` 状态的调研任务重新推入队列执行，并通过外部 AI 重新评估后向用户推送更新后的报告。
- **FR-015**: System MUST 提供可配置的隐私策略选项，采用独立的开关控制以下子项（默认均为关闭/受限）：(1) 是否允许在必要时向外部 AI 发送完整知识内容；(2) 是否允许调用外部网络搜索；(3) 是否允许上传运行日志到外部服务；默认状态下仅允许发送本地检索后的相关片段及必要摘要，用户可显式开启各子项以放宽限制。
- **FR-016**: System MUST 保存知识条目的修改历史版本；默认保留全部版本，不自动清理；System MUST 允许用户显式配置保留策略（如按版本数量、时间或磁盘空间阈值自动清理），配置生效后才启动自动清理。
- **FR-017**: System MUST 为本地 Web 访问提供单用户密码鉴权；System MUST 对本地原始数据文件进行加密存储，采用 AES-256-GCM 加密并配合 Argon2id 从用户密码派生解密密钥，未通过鉴权时不得以任何形式解密或访问原始数据。鉴权采用短期 JWT（默认 24 小时有效期），服务重启或 Token 过期后用户必须重新登录；System SHOULD 支持"记住我"选项以延长至 7 天。
- **FR-018**: System MUST 不绑定特定外部 AI 服务商，外部聊天和嵌入服务通过通用配置接入任意兼容 OpenAI API 的端点。
- **FR-019**: System MUST 采用 SQLite 作为本地主存储，结合 sqlite-vec 作为全文/向量扩展管理结构化数据、索引及嵌入；原始媒体文件 MUST 存储于本地文件系统，按知识条目 ID 分两级目录组织（如 `files/AB/CD/<item-id>/original.jpg`），并通过路径与元数据关联。
- **FR-020**: System MUST 支持为每条知识条目分配一个或多个标签（多标签），以便用户进行分类组织和快速筛选。
- **FR-021**: System MUST 允许用户从知识库中删除知识条目，删除操作采用软删除（仅标记为已删除），保留原始数据及历史引用关系，确保历史对话和调研报告中仍可正常引用并展示该来源。
- **FR-022**: System MUST 在外部 AI/搜索服务请求失败时实现自动重试机制：默认最多重试 3 次并采用指数退避；最终仍失败时必须向用户明确提示；System MUST 允许用户通过配置调整重试次数与超时时间。
- **FR-023**: System MUST 支持用户同时提交多个调研任务，任务并发数以配置的外部 API 调用配额为上限，超出上限的任务自动进入队列等待执行，并向用户展示任务队列状态。
- **FR-024**: System MUST 记录应用错误日志和关键用户操作日志（如知识增删改、调研任务触发与完成），用于故障排查和基本审计；日志 MUST 存储于本地文件系统，支持用户配置日志保留策略。
- **FR-025**: System MUST 在用户忘记单用户密码时明确提示本地加密数据不可恢复，并提供系统重新初始化与导入已有备份的引导流程。
- **FR-026**: System SHOULD 在知识库总容量超过用户配置的磁盘阈值时，自动对旧媒体文件进行归档压缩（如 gzip），以维持本地磁盘读写性能。

### Key Entities *(include if feature involves data)*

- **KnowledgeItem（知识条目）**: 代表用户存入知识库的一条内容单元，以 UUID 作为唯一标识，包含标题、原始内容、来源类型、标签（多标签）、创建时间、置信度评分及历史版本记录等属性；MVP 阶段不检测重复内容。
- **KnowledgeBase（知识库）**: 用户的个人知识集合，包含多条 KnowledgeItem，支持检索和浏览。
- **Conversation（对话）**: 用户与系统之间的问答交互记录，包含用户提问、系统回答及引用的知识来源；历史对话用于推导用户画像。
- **UserProfile（用户画像）**: 基于用户对话和交互历史推导出的领域偏好、知识水平及学习需求，用于个性化调研和报告生成。
- **ResearchReport（调研报告）**: 基于用户指定的调研主题生成的结构化文档，包含主题、章节内容、参考来源列表。
- **ConfidenceEvaluation（置信度评估）**: 针对单个 KnowledgeItem 特定版本的评估结果，包含评分等级、评估方法说明、生成时间及关联的版本标识。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能够在 2 分钟内完成一条新知识的添加操作（从输入到保存成功）。
- **SC-002**: 90% 的对话查询能够获得与知识库内容相关且准确的回答。
- **SC-003**: 调研报告在用户提交主题后的 5 分钟内可用（异步处理场景下指报告生成完成并可查看）。
- **SC-004**: 用户能够在知识库列表或详情页中直观识别知识条目的置信度等级（高/中/低）。
- **SC-005**: 系统能够在知识库包含至少 100 条知识时，查询和展示响应时间仍保持 ≤2 秒（端到端）。

## Clarifications

### Session 2026-04-05

- Q: 外部调研的搜索源集成方式是什么？ → A: C+D：优先尝试 LLM 自带搜索能力，不支持时回退到用户配置的独立搜索 API，若仍不可用则使用系统内置轻量 HTTP 爬虫直接抓取网页。
- Q: 知识条目内容发生修改时，版本生成与置信度重新评估的触发策略是什么？ → A: C - 任何 `content` 变更均生成新版本，但仅当差异 >20% 或被用户手动请求时才触发置信度重新评估。
- Q: 本地单用户应用的会话与 Token 策略应如何定义？ → A: B - 采用短期 JWT（默认 24 小时有效期），服务重启或 Token 过期后需重新登录；支持"记住我"延长至 7 天。
- Q: 外部服务恢复后，因外部服务失败而暂停的调研任务等待状态应如何定义？ → A: B - 引入独立的 `pending_recheck` 状态，外部服务恢复后仅将该状态任务重新推入 `queued` 执行。
- Q: `privacy_settings.allow_web_search`、`llm_config.enable_search` 与 `search_config` 在控制外部搜索启用时的优先级和职责边界应如何定义？ → A: A - `allow_web_search` 为总开关，关闭时禁止任何外部网络搜索；开启时按 FR-006a 优先级尝试，`llm_config.enable_search` 仅控制是否优先尝试 LLM 自带搜索。
- Q: 对话查询中的 RAG 检索采用向量搜索和全文搜索并行执行后，混合排序与结果合并策略应如何定义？ → A: B - 并行召回 + 去重合并：向量搜索召回 Top-15，FTS5 召回 Top-15，去重合并后按相似度/相关度分数加权排序，取 Top-10 作为上下文。
- Q: SQLite 全文/向量扩展的具体选型是什么？ → A: A - 使用 sqlite-vec。
- Q: 历史版本与置信度的关联方式是什么？ → A: A - 每个历史版本独立保存其置信度评估结果；修改生成新版本时自动重新评估（若内容变化 >20%），旧版本置信度保持不变。
- Q: 隐私策略各模式的具体行为定义是什么？ → A: C - 以开关形式定义，用户独立控制是否允许发送完整知识、是否允许外部搜索、是否允许日志上传等子项。
- Q: 知识库导出/导入时嵌入向量的跨系统兼容性策略是什么？ → A: A - 导出包不包含嵌入向量，仅保留 embedding 模型标识；导入时按需自动重新计算。

- Q: 用户同时提交多个调研任务时的并发策略是什么？ → A: 不限制用户发起数量，以外部 API 调用配额为准进行并发控制，超出配额的任务进入队列等待执行。

## Assumptions

- 目标用户为个人使用者，MVP 阶段以单用户个人知识库为主；明确排除多用户协作、移动端原生应用。第三方生产力工具集成不在 MVP 必选范围，但保留未来扩展可能。
- 系统以本地服务形式运行，用户通过浏览器访问，单用户登录即可使用。
- 用户具备基本的互联网接入条件，以支持调研和置信度验证所需的外部信息检索。
- 知识输入支持文本、常见文档格式、网页链接以及图片、代码、视频、音频等多种媒体类型。
- 置信度评估结果作为一种参考提示，不要求达到学术论文级别的严格事实核查标准。
