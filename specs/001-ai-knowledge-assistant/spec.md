# Feature Specification: AI 知识管理助手

**Feature Branch**: `001-ai-knowledge-assistant`  
**Created**: 2026-04-04  
**Status**: Draft  
**Input**: User description: "需要实现一个基于ai驱动的知识管理助手，支持为用户管理知识(类似知识库)，支持对话查询知识，支持基于用户需求调研领域知识，形成调研报告，支持基于网络或者常识评估知识的置信度。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 添加并管理个人知识 (Priority: P1)

用户可以将日常收集到的文档、笔记、网页链接等内容整理进一个统一的知识库，方便后续查找和使用。

**Why this priority**: 知识库管理是整个产品的核心基础能力。没有可靠的知识入库和管理，后续的查询、调研等功能都无从谈起。

**Independent Test**: 用户能够独立上传一段文本并保存，随后在知识库列表中查看到该内容，验证其标题和摘要是否正确展示。

**Acceptance Scenarios**:

1. **Given** 用户进入知识库管理页面，**When** 用户输入一段文本并确认保存，**Then** 系统将该内容保存为一条知识，并在知识库列表中可见。
2. **Given** 用户已保存多条知识，**When** 用户搜索关键词，**Then** 系统返回标题或内容匹配的相关知识列表。

---

### User Story 2 - 对话式查询知识 (Priority: P1)

用户通过自然语言与助手进行对话，系统基于用户个人知识库中的内容给出准确回答，并标注信息来源。

**Why this priority**: 对话查询是用户获取知识价值的最直接方式，是驱动用户持续积累知识的核心动机。

**Independent Test**: 用户在知识库中已保存一条关于某主题的内容后，用自然语言提问该主题，系统给出的回答必须引用该内容。

**Acceptance Scenarios**:

1. **Given** 知识库中已存在相关知识，**When** 用户以自然语言提问，**Then** 系统基于库中内容生成回答，并指明参考来源。
2. **Given** 知识库中不存在与用户问题相关的内容，**When** 用户提问，**Then** 系统告知用户当前知识库中未找到相关信息，而非编造答案。

---

### User Story 3 - 生成领域调研报告 (Priority: P2)

用户输入一个调研主题，系统以异步方式检索网络信息并综合生成一份结构化的调研报告。用户在调研过程中可查看实时进度和阶段性输出；当系统遇到需决策的问题时会暂停并向用户提问，待用户回复后继续生成，最终帮助用户快速了解陌生领域。

**Why this priority**: 相比被动查询已有知识，主动调研能够扩展用户知识边界，是产品差异化的重要能力。

**Independent Test**: 用户提交一个明确的调研主题（如"低空经济政策分析"），在调研过程中能查看进度；当系统遇到分支决策时页面向用户提问，用户给出决策后系统继续生成，最终返回一份包含背景、现状、趋势、结论等章节的报告。

**Acceptance Scenarios**:

1. **Given** 用户提交调研主题和范围描述，**When** 系统开始处理，**Then** 用户可在页面上查看调研进度及阶段性输出摘要。
2. **Given** 调研过程中系统遇到需要用户决策的节点，**When** 系统暂停并向用户展示决策选项，**Then** 用户做出选择后调研流程继续执行。
3. **Given** 调研完成后，**When** 用户选择将报告保存到知识库，**Then** 系统以知识条目形式存储该报告并支持后续对话查询。

---

### User Story 4 - 知识置信度评估 (Priority: P2)

系统对每个知识条目进行置信度评估，帮助用户识别可靠信息和可能存在疑问的内容。

**Why this priority**: 置信度评估能够提升用户对知识库内容的信任感，但属于增强型体验，可在核心功能稳定后逐步完善。

**Independent Test**: 用户上传一条带有明显事实性陈述的知识后，系统显示该知识的置信度评分及评估依据（如网络验证结果或常识一致性说明）。

**Acceptance Scenarios**:

1. **Given** 系统接收到新知识，**When** 置信度评估完成后，**Then** 该知识条目显示一个可视化的置信度评分（如高/中/低或百分比）。
2. **Given** 用户查看某条知识的详情，**When** 用户查看置信度信息，**Then** 系统展示评估依据的简要说明。

---

### Edge Cases

- 用户上传的知识内容为空或仅有无意义字符时，系统如何提示？
- 当调研主题过于宽泛或模糊时，系统如何请求用户补充范围？
- 本地服务未启动时，用户通过浏览器访问应收到明确的服务不可用提示。
- 网络不可用或外部搜索源受限时，本地知识库管理和查询保持可用；若配置了本地模型，调研可降级至本地模型执行并提示用户处于降级模式；外部服务恢复后，系统重新通过外部 AI 评估并推送更新后的调研报告。
- 同一条知识被多次修改后，历史版本和置信度如何追踪？
- 导入备份压缩包时文件损坏或包含不支持格式的知识，系统如何安全跳过并提示用户？
- 知识库总容量随多媒体文件增长而扩大时，系统如何保证本地磁盘读写性能不显著下降？
- 用户忘记密码时，本地加密的原始数据无法被恢复访问，系统应如何明确提示并引导用户？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 允许用户以文本、文件或网页链接的形式添加知识到个人知识库，支持图片、代码、视频、音频等多种格式，单文件大小上限为 1GB。System MUST 自动将非文本媒体内容（如图片 OCR、音视频转录、文档全文提取）转换为可搜索文本，以便支持检索和对话查询；当自动提取失败时，System MUST 保存原始文件并标记提取失败状态，同时尝试利用文件名、扩展名等元数据进行极简索引，并向用户明确提示该文件暂不支持内容级搜索与对话引用。
- **FR-002**: System MUST 将用户添加的知识组织为可检索、可浏览的知识条目列表。
- **FR-003**: System MUST 支持自然语言对话，用户可通过提问查询个人知识库中的内容；System MUST 持久化保存完整对话历史，允许用户查看会话列表并继续之前的话题。
- **FR-004**: System MUST 在回答问题时引用所依据的知识来源，无法找到依据时必须明确告知用户。
- **FR-005**: System MUST 基于用户对话历史提取用户的领域偏好和知识水平，在生成调研报告和回答问题时进行个性化调整（如已知领域可减少基础说明，未知领域需补充背景介绍）。
- **FR-006**: System MUST 允许用户提交一个调研主题，并在合理时间内生成结构化的调研报告；System MUST 通过 Server-Sent Events (SSE) 向页面实时推送调研进度及阶段性输出摘要。
- **FR-007**: 调研报告 MUST 综合多个来源的信息，包含背景、关键观点、趋势分析及结论等结构化章节。
- **FR-008**: System MUST 在知识入库/更新时自动对每条知识进行置信度评分，并在界面上可视化展示；后续仅在内容发生显著变化（文本差异比例 >20%）或被用户主动请求时重新评估；System MUST 提供手动重新评估入口，允许用户随时触发。
- **FR-009**: 置信度评估 MUST 借助外部网络信息或常识推理进行验证，并给出简要依据。
- **FR-010**: System MUST 提供知识库导出功能，将当前用户的全部知识及相关元数据打包为 ZIP 压缩文件；压缩包内必须包含一份 JSON 元数据文件（描述知识条目、版本记录及目录结构），并将原始媒体附件按原路径或目录结构存放。
- **FR-011**: System MUST 支持通过导入已导出的 ZIP 压缩文件，在新系统环境中恢复知识库，实现数据迁移；导入过程必须校验 JSON 元数据，并安全跳过损坏或不支持的文件。
- **FR-012**: 当外部 AI/搜索服务不可用时，System MUST 保持本地知识库管理和对话查询功能可用，并向用户明确提示调研和置信度评估能力受限。
- **FR-013**: 若用户配置了本地模型，System SHOULD 在外部服务不可用时将调研功能降级至本地模型执行，并在界面上提示用户当前处于降级模式。
- **FR-014**: 当外部服务从不可用恢复为可用后，System SHOULD 自动通过外部 AI 重新评估相关调研任务，并向用户推送更新后的报告。
- **FR-015**: System MUST 提供可配置的隐私策略选项，默认采用"本地优先"模式（仅向外部 AI 发送检索后的相关片段及必要摘要），并允许用户显式调整为更严格或更宽松的模式。
- **FR-016**: System MUST 保存知识条目的修改历史版本，并允许用户配置保留策略（如按版本数量、时间或磁盘空间阈值自动清理）。
- **FR-017**: System MUST 为本地 Web 访问提供单用户密码鉴权，用户每次访问系统前都必须通过身份验证；System MUST 对本地原始数据文件进行加密存储，未通过鉴权时不得以任何形式解密或访问原始数据。
- **FR-018**: System MUST 不绑定特定外部 AI 服务商，外部聊天和嵌入服务通过通用配置接入任意兼容 OpenAI API 的端点。
- **FR-019**: System MUST 采用 SQLite 作为本地主存储，结合轻量全文/向量扩展管理结构化数据、索引及嵌入；原始媒体文件 MUST 存储于本地文件系统，并通过路径与元数据关联。
- **FR-020**: System MUST 支持为每条知识条目分配一个或多个标签（多标签），以便用户进行分类组织和快速筛选。
- **FR-021**: System MUST 允许用户从知识库中删除知识条目，删除操作采用软删除（仅标记为已删除），保留原始数据及历史引用关系，确保历史对话和调研报告中仍可正常引用并展示该来源。
- **FR-022**: System MUST 在外部 AI/搜索服务请求失败时实现自动重试机制：默认最多重试 3 次并采用指数退避；最终仍失败时必须向用户明确提示；System MUST 允许用户通过配置调整重试次数与超时时间。
- **FR-023**: System MUST 支持用户同时提交多个调研任务，任务并发数以配置的外部 API 调用配额为上限，超出上限的任务自动进入队列等待执行，并向用户展示任务队列状态。

### Key Entities *(include if feature involves data)*

- **KnowledgeItem（知识条目）**: 代表用户存入知识库的一条内容单元，包含标题、原始内容、来源类型、标签（多标签）、创建时间、置信度评分及历史版本记录等属性。
- **KnowledgeBase（知识库）**: 用户的个人知识集合，包含多条 KnowledgeItem，支持检索和浏览。
- **Conversation（对话）**: 用户与系统之间的问答交互记录，包含用户提问、系统回答及引用的知识来源；历史对话用于推导用户画像。
- **UserProfile（用户画像）**: 基于用户对话和交互历史推导出的领域偏好、知识水平及学习需求，用于个性化调研和报告生成。
- **ResearchReport（调研报告）**: 基于用户指定的调研主题生成的结构化文档，包含主题、章节内容、参考来源列表。
- **ConfidenceEvaluation（置信度评估）**: 针对单个 KnowledgeItem 的评估结果，包含评分等级、评估方法说明、生成时间。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能够在 2 分钟内完成一条新知识的添加操作（从输入到保存成功）。
- **SC-002**: 90% 的对话查询能够获得与知识库内容相关且准确的回答。
- **SC-003**: 调研报告在用户提交主题后的 5 分钟内可用（异步处理场景下指报告生成完成并可查看）。
- **SC-004**: 用户能够在知识库列表或详情页中直观识别知识条目的置信度等级（高/中/低）。
- **SC-005**: 系统能够在知识库包含至少 100 条知识时，查询和展示响应时间仍保持 ≤2 秒（端到端）。

## Clarifications

### Session 2026-04-04

- Q: 产品的部署形态和访问方式是什么？ → A: 本地服务 + 浏览器访问，个人应用，web单用户登录即可。
- Q: 单条知识大小上限和整个知识库总容量预期是多少？ → A: 单文件 ≤1GB，总容量无明确上限；支持图片、代码、视频、音频等多媒体；须有备份迁移功能（导出压缩包 + 跨系统导入）。
- Q: 调研报告生成应采用同步还是异步方式？ → A: 异步；页面需展示调研进度和实时输出，遇到需决策的节点时向用户提问并等待决策。
- Q: 当外部 AI 或搜索服务不可用时系统的降级策略是什么？ → A: 基础功能保持可用；若配置了本地模型，调研可降级使用本地模型执行，同时向用户明确提示当前处于降级模式；外部服务恢复后，系统自动通过外部 AI 重新评估调研并推送更新后的报告。
- Q: 用户知识库内容在调用外部 AI 服务时，隐私和数据安全策略是什么？ → A: 用户可配置策略，默认本地优先（仅在本地检索片段后发送相关片段；调研和置信度评估按需发送必要摘要），用户也可自行调整为更严格或更宽松的模式。
- Q: 知识条目是否支持修改历史版本追踪？如支持，版本保留策略是什么？ → A: 保留所有版本，支持用户配置保留策略（如按版本数、时间或磁盘空间阈值自动清理）。
- Q: 知识置信度评估应在何时触发？ → A: 首次入库/更新时自动触发；后续仅在内容发生显著变化或被用户主动请求时重新评估。
- Q: 对于图片、视频、音频等非文本知识，系统应如何支持内容检索和对话查询？ → A: 系统必须自动提取所有媒体中的可搜索文本（图片 OCR、音视频转录、文档全文提取），原始文件作为附件。
- Q: 对话式查询是否需持久化保存历史对话记录，并支持用户查看和继续之前的会话？ → A: 需要持久化保存完整历史，支持会话列表、查看和继续之前的话题；系统应基于对话历史提取用户的偏好和知识水平，在调研和生成报告时进行个性化调整（如熟悉的领域可省略基础知识，不熟悉的领域需补充背景说明）。
- Q: 知识库导出/导入的压缩包应采用什么具体格式？ → A: ZIP 压缩包，内部包含 JSON 元数据文件和原始媒体附件。
- Q: 本地单用户 Web 应用是否需要身份验证，原始数据如何保护？ → A: 每次访问前都需单用户密码登录；原始数据文件在无密码时不允许解密或访问，必须采用加密存储。
- Q: 知识库达到 100 条时，查询/展示响应时间的可接受上限是多少？ → A: ≤2 秒。
- Q: 知识内容发生"显著变化"时应如何量化定义以重新触发置信度评估？ → A: 基于文本差异比例，编辑距离/字数变化超过 20% 时自动重新评估；同时系统必须支持用户手动触发重新评估，即使编辑幅度小于 20%。
- Q: 当系统对非文本媒体的自动文本提取失败时，应如何处理该文件？ → A: 保存原始文件但标记提取失败，提示用户该文件暂不支持搜索/对话引用；同时尝试利用文件名、扩展名等元数据进行极简索引，以便用户可通过名称搜索。
- Q: MVP 阶段明确排除的功能范围是什么？ → A: 排除多用户协作、移动端原生应用；保留第三方生产力工具集成（作为未来扩展方向，非 MVP 必选）。
- Q: 外部 AI/搜索服务是否绑定特定供应商？ → A: 不指定具体服务商，仅要求通过配置可切换任意兼容 OpenAI API 的端点。
- Q: 异步调研进度和阶段性输出采用什么实时推送机制？ → A: Server-Sent Events (SSE)。
- Q: 本地持久化存储应采用什么技术方案？ → A: SQLite 为主存储 + 轻量全文/向量扩展，多媒体原始文件存本地文件系统。
- Q: 知识条目的组织方式采用文件夹层级还是标签系统？ → A: 支持标签（多标签），为每条知识条目分配一个或多个标签用于分类和快速筛选。
- Q: 用户是否可以从知识库中删除知识？删除后历史对话与调研报告中的引用如何处理？ → A: 支持删除，采用软删除（标记为已删除但保留数据），历史对话和报告中仍可正常引用显示。
- Q: 当外部 AI/搜索服务请求失败时，系统应采取什么重试与错误处理策略？ → A: 默认自动重试最多 3 次（指数退避），失败后明确提示用户；支持用户配置重试次数和超时时间。
- Q: 用户同时提交多个调研任务时的并发策略是什么？ → A: 不限制用户发起数量，以外部 API 调用配额为准进行并发控制，超出配额的任务进入队列等待执行。

## Assumptions

- 目标用户为个人使用者，MVP 阶段以单用户个人知识库为主；明确排除多用户协作、移动端原生应用。第三方生产力工具集成不在 MVP 必选范围，但保留未来扩展可能。
- 系统以本地服务形式运行，用户通过浏览器访问，单用户登录即可使用。
- 用户具备基本的互联网接入条件，以支持调研和置信度验证所需的外部信息检索。
- 知识输入支持文本、常见文档格式、网页链接以及图片、代码、视频、音频等多种媒体类型。
- 置信度评估结果作为一种参考提示，不要求达到学术论文级别的严格事实核查标准。
