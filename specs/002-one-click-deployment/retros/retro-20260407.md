## What Could Improve 🔺

1. **Implementation-to-contract drift**: `deploy.py` 最初的 `--host 0.0.0.0` 与 `contracts/cli-contract.md` 约定的 `127.0.0.1` 不一致，直到 QA 才被发现。说明编码阶段缺少对契约的主动核对。
2. **Underestimating system-level detection**: US2 的冲突检测任务（T012/T013）在初步实现时只做了端口占用检查，未考虑到“端口被占用 ≠ 本应用正在运行”的场景，导致 FR-010 最初未满足。
3. **CLI argument handling gap**: `deploy.py --help` 最初会忽略未识别参数并直接启动服务，这说明边缘场景（参数解析）在 spec 和 plan 阶段没有被显式覆盖。
4. **No formal code review**: 本周期没有单独的 code review 记录，QA 承担了主要的质量守门角色。虽然 QA 很高效，但一些明显的问题（如 host 参数、参数解析）本可以在 review 阶段更早发现。

## Suggested Constitution Updates

基于本次 retrospective，建议在项目 `constitution.md` 中新增以下原则：

1. **契约优先于实现**: 当代码与契约（contract）发生冲突时，默认优先修正代码。若必须修改契约，应同步更新所有相关文档并记录变更理由。
   _Learned from: 002-one-click-deployment 中 host 参数与契约不一致，导致 QA 返工。_

2. **系统级探测需要双重验证**: 任何涉及进程、端口、文件锁等系统级状态检测的实现，应同时提供“正向验证”和“负向验证”场景，避免过度简化。
   _Learned from: FR-010 最初仅检测端口占用，未检测到同应用多实例启动的风险。_
