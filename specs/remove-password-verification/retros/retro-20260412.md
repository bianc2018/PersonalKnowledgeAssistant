## 2.  What Went Well

### 2.1. 流程纪律优秀
- **每个阶段都留痕**：从 `problem-statement.md` 到 `verify-report.md`，每个 Product Forge 阶段都有可审计的文档产出。
- **Gate 决策清晰**：`.forge-status.yml` 完整记录了每个 gate 的批准时间、决策理由和通过条件。
- **小步提交**：backend、frontend、CLI、tests 分别独立提交，便于回滚和审查。

### 2.2. 架构决策前置，避免实现期返工
- 在 **Phase 1 Research** 中，通过竞品分析（Jupyter、Ollama、Obsidian、Home Assistant 等 8 个工具）快速确认了「可选密码」是本地单用户工具的主流模式。
- 在 **Phase 2 Product Spec** 中，明确了一旦选定不可切换（避免附件解密失败），这个设计约束在后续实现中被严格遵守，避免了运行时状态机的复杂性。

### 2.3. 测试策略有效
- **向后兼容测试**：通过旧 fixture 显式传入 `password_enabled=True`，确保原有密码模式未被破坏。
- **新增路径覆盖**：`no_auth_client` fixture 直接验证了无密码初始化 → 自动登录 → 附件加解密的完整链路。
- **回归测试**：发现了 `InitRequest` 中 `min_length=8` 的意外移除，并在第二轮提交中修复。

### 2.4. 单文件改动小，审查友好
- 虽然 feature 总 LOC 接近 3k，但其中约 2/3 是 specs/ 目录中的文档和研究材料。
- 核心代码变更分散在 8 个代码文件中，每个文件的职责单一，Code Review 时容易聚焦。

## 3. What Could Be Improved

### 3.1. 安全设计在 spec 阶段未充分暴露
- **SC1 (Critical)**：`get_no_auth_master_key()` 使用了未加盐的 `SHA256(SECRET_KEY)`，这个问题直到 Code Review 才被标记。
- **教训**：涉及加密的设计应该在 Plan 阶段就引入安全审查检查清单（如「主密钥派生是否使用 KDF？」「盐的存储策略是什么？」）。
- **改进建议**：在 Plan 模板中增加一个「Security Checklist」小节，强制要求在编码前回答 3-5 个安全相关问题。

### 3.2. Pydantic 默认值理解有盲区
- `InitRequest.password: str | None = None` 的写法在第一次提交时丢失了 `min_length=8` 校验，导致既有回归测试失败。
- **教训**：当把 `Field(..., min_length=8)` 改为可选字段时，需要注意 `Field(default=None, min_length=8)` 的写法。
- **改进建议**：在修改 Pydantic 模型时，运行相关测试子集 `pytest tests/integration/test_auth_system.py` 作为预提交检查。

### 3.3. 环境相关的 flaky 测试消耗了验证时间
- `tests/integration/test_deploy_daemon.py::test_daemon_lifecycle` 在当前沙箱环境下始终失败（`RemoteDisconnected`），经验证是环境基础设施问题而非回归。
- **教训**：约 10 分钟被用于排查这个已知/预存在的失败。
- **改进建议**：建立一个「已知 flaky 测试」清单，并在 CI 或本地测试运行前自动跳过这些测试（通过 pytest marker 或显式 exclude list）。

### 3.4. Code Review 与实现存在时间重叠
- 由于上下文压缩，部分实现是在对话中断后恢复的，导致 Code Review 阶段实际上由 Plan agent 代为执行，而非独立的 review pass。
- **教训**：长 feature 的实现/审查流程在单会话内完成更可靠；跨会话恢复时，状态重建成本高。
- **改进建议**：对于能在 2-3 小时内完成的 feature，优先一次性完成实现+审查；对于更大的 feature，使用 checkpoint 保存中间状态。

## 5. Patterns Observed

### 5.1. 文档先行 vs 代码先行
- 这个 feature 遵循了严格的**文档先行**模式：spec → plan → tasks → 实现。
- 结果是实现阶段几乎没有架构级别的犹豫，每个文件的修改边界都很清晰。

### 5.2. 单领域变更集中
- 所有后端 auth 变更集中在 `src/auth/*` 和 `src/system/*`。
- 所有前端变更集中在 `src/web/static/js/pages/init.js` 和 `app.js`。
- 没有跨多个无关子系统的「散弹枪式」修改。

### 5.3. 向后兼容作为第一优先级
- `password_enabled INTEGER NOT NULL DEFAULT 1` 的 schema 设计确保了旧数据零迁移成本。
- 密码模式的全部回归测试未被破坏。

## 6. Action Items for Next Feature

1. **在 Plan 阶段加入安全审查检查清单**
   - 目标：避免 Code Review 时才暴露加密设计缺陷。
   - 具体：在 `plan.md` 中增加「Security Checklist」小节，包含 KDF/Salt、输入验证、认证绕过场景等 5 个问题。

2. **建立 flaky 测试排除清单**
   - 目标：减少环境噪音对验证时间的消耗。
   - 具体：创建 `pytest.ini` 中的 `markers` 或 `deselect` 配置，将 `test_daemon_lifecycle` 标记为 `flaky` 并在常规 CI 中跳过。

3. **Pydantic 模型变更预提交检查**
   - 目标：避免验证器意外丢失。
   - 具体：修改 `Field(...)` 为可选字段时，强制使用 `Field(default=None, ...)` 而非裸 `= None`。

4. **考虑将 no-auth master key 升级为 HKDF**
   - 目标：修复 SC1 Critical finding。
   - 具体：在下一个 patch 版本中将 `hashlib.sha256(settings.secret_key.encode()).digest()` 替换为 `cryptography.hazmat.primitives.kdf.hkdf.HKDF`。
