## What Went Well 🎉

1. **极短交付周期**: 从进入 Implement 到 Release Readiness 仅约 1.5 天，29 项任务全部完成，体现了高度聚焦的执行效率。
2. **零 Node 架构验证成功**: 仅用静态文件 + CDN 即搭建了功能完整的 SPA，避免了构建工具链的额外复杂度，对本地部署极友好。
3. **fetch + ReadableStream SSE 方案跑通**: 在不修改后端的前提下，前端通过手动 SSE 解析成功消费了 Bearer 保护的 Chat/Research 流端点，保持了前后端解耦。
4. **Code Review 发现并修复了关键缺陷**: 虽然进入 Code Review 较晚，但一轮 review 就精准定位了 P0 SSE、P1 XSS、P2 Router 三类问题，且修复质量高、无回归。
5. **后端测试稳定**: 新增附件下载集成测试与既有测试套件全部通过，为前端的“手动 E2E 受沙盒限制”提供了有力的质量兜底。

## What Could Improve 🔺

1. **P0 SSE 缺陷未在实现阶段自测发现**: Chat 的 SSE 流在 `T-017` 完成后实际上完全不可用（`sse.js` 默认 GET + 不解析 `event:`，`chat.js` 双重调用 `apiPost` 与 `createSSEStream`），直到 Code Review 才暴露。说明实现阶段缺少对 SSE 端到端的基础自测。
2. **系统性的 XSS 风险**: `knowledge.js`、`research.js`、`dashboard.js` 中大量模板字符串直接拼接了后端返回的用户输入，未做任何转义。这是实现阶段的模式问题，而非单点疏忽。
3. **前端测试覆盖率薄弱**: 除后端 API 集成测试外，前端没有任何自动化测试（单元/E2E）。响应式、键盘导航、401 拦截、XSS 边界等完全依赖手工 QA 和代码审查。
4. **手动 E2E 被沙盒环境阻断**: 由于 `http_proxy` 导致 localhost 请求被代理丢弃，无法运行 live browser E2E，只能依赖 pytest + 代码走读验证。未来若要在 CI 中保障前端链路，需要解决代理/无头浏览器环境或引入 Playwright 等自动化方案。

## Suggested Constitution Updates

基于本次 retrospective，建议在项目 `constitution.md` 中新增以下原则：

1. **流式与二进制链路必须实现后立即自测**: 任何涉及 SSE、文件下载/上传、导出导入的功能，开发者在提交前必须在真实浏览器或 curl 中完成至少一次端到端验证。
   _Learned from: 003-web-frontend 的 Chat SSE P0 缺陷直到 Code Review 才被发现，原因是编码者未在浏览器中实际发送一条消息验证流式输出。_

2. **模板字符串中的用户输入默认不信任**: 所有拼接进 HTML 模板字符串的动态数据（来自后端、URL 参数、用户输入）必须使用转义函数（如 `escapeHtml`），除非经过明确的安全审计。
   _Learned from: 003-web-frontend 的知识库、调研、Dashboard 页面中大量未转义的模板插值构成了系统性的 XSS 风险面。_
